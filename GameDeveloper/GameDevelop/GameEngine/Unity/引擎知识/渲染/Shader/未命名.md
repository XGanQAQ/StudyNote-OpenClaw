# 🎯 Unity 渲染系统核心架构笔记

---

## 一、渲染体系总览

在 Unity Scriptable Render Pipeline（SRP）体系中，渲染流程被拆分为多个可控制模块：

```
Renderer
 ├── RenderFeature（注册阶段）
 │        └── RenderPass（执行阶段）
 │
 └── Volume System（参数阶段）
```

可以理解为：

|模块|职责|
|---|---|
|RenderPass|真正执行渲染|
|RenderFeature|控制何时执行|
|Volume|控制执行参数|

---

# 二、RenderPass

## 定义

**RenderPass = 一段 GPU 渲染指令执行单元**

它负责：

- 设置 RenderTarget
    
- 调用 DrawRenderers
    
- Blit
    
- DispatchCompute
    
- 设置 Shader 参数
    

---

## 本质定位

> RenderPass 是渲染管线最小执行单位

没有它，就不会产生任何画面变化。

---

## 生命周期（重要）

RenderPass 每帧执行流程：

```
OnCameraSetup
Execute
OnCameraCleanup
```

---

## 常见用途

- 自定义后处理
    
- 描边
    
- 深度预处理
    
- ShadowMask
    
- 自定义光照
    
- Object Mask
    

---

---

# 三、RenderFeature

## 定义

**RenderFeature = RenderPass 的注册器**

它的作用只有一个：

> 把 RenderPass 插入到渲染流程指定阶段

---

## 内部职责

```
Create()
AddRenderPasses()
Dispose()
```

最核心函数：

```
AddRenderPasses(renderer, ref renderingData)
```

这里决定：

```
把哪个Pass
插到哪个阶段
```

---

## 插入阶段（极重要）

可选执行时机：

```
BeforeRendering
AfterRenderingOpaques
BeforeRenderingTransparents
AfterRenderingPostProcessing
```

---

📌 本质理解

> RenderFeature 不做渲染  
> 只是决定 Pass 是否执行 + 在什么时候执行

---

# 四、Volume 系统

---

## 定义

**Volume = 渲染参数调度系统**

---

## 解决问题

它解决的是：

```
多个来源参数 → 如何混合为最终值
```

而不是渲染。

---

---

## Volume 组成结构

```
Volume
 └── Profile
       └── VolumeComponent
              └── Parameter
```

例如：

```
Bloom
 ├ intensity
 ├ threshold
 └ scatter
```

---

---

## 运行流程（每帧）

```
相机更新
    ↓
收集所有Volume
    ↓
按权重混合参数
    ↓
生成最终VolumeStack
    ↓
RenderPass读取
```

---

📌 核心结论

> Volume 本身不会产生画面  
> 它只是给 Pass 提供参数

---

---

# 五、三者关系总图（最重要理解）

```
Volume → 提供数据
Feature → 注册执行点
Pass → 执行渲染
```

或者一句话版本：

> Volume 决定数值  
> Feature 决定时机  
> Pass 决定画什么

---

---

# 六、真实渲染执行顺序（URP简化）

```
Camera Start
   ↓
Volume计算参数
   ↓
Renderer 收集 Feature
   ↓
Feature 注册 Pass
   ↓
按Event排序 Pass
   ↓
逐个执行 Pass
   ↓
Camera End
```

# 七、常见误区（非常重要）

---

## ❌ 误区1

**“添加Volume Override就等于添加效果”**

错。

正确：

```
Override只是参数
必须有Pass读取才有效果
```

---

---

## ❌ 误区2

**“RenderFeature会渲染画面”**

错。

RenderFeature 只是注册器。

---

---

## ❌ 误区3

**“Volume只能做后处理”**

错。

Volume 可控制任何渲染参数：

- 阴影强度
    
- 光照强度
    
- Shader变量
    
- 自定义Pass参数
    

---

---

## ❌ 误区4

**“RenderPass执行顺序固定”**

错。

执行顺序由：

```
RenderPassEvent + enqueue顺序
```

决定。

# 八、架构级理解（高级开发者视角）

Unity SRP 实际采用的是经典架构模式：

```
数据驱动渲染
```

拆解如下：

|系统|角色|
|---|---|
|Volume|配置层|
|Renderer|调度层|
|Pass|执行层|

---

这属于标准三层结构思想：

```
配置层
逻辑层
执行层
```

优点：

- 高扩展
    
- 高解耦
    
- 高可维护
    

---

---

# 九、什么时候用哪个？

|需求|用什么|
|---|---|
|自定义绘制|RenderPass|
|控制执行时机|RenderFeature|
|控制参数|Volume|
|做完整效果|Feature + Pass + Volume|

---

---

# 十、面试级总结（推荐背）

> Unity SRP 渲染架构采用模块化设计：RenderPass 负责具体渲染执行，RenderFeature 负责将 Pass 插入到渲染流程指定阶段，而 Volume 系统负责在渲染前统一计算并混合最终参数，从而实现渲染逻辑与渲染数据的完全解耦。
