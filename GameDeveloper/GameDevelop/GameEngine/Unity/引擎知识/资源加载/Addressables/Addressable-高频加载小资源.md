Addressables 最耗性能的不是资源加载，而是：
> Handle对象创建 + Task调度 + 回调执行
> 这是 高频加载小资源会慢 的原因

## 为什么？

> ❗ 并不是所有情况下 Addressables 最耗性能的都是 Handle + Task + 回调  
> ✅ 只有在**资源本体加载成本很低**时，它们才会成为主要开销

也就是说：

```
大资源 → IO/解压最耗时
小资源高频加载 → 调度系统最耗时
```

下面我帮你彻底拆解这个问题的底层原理。

---

# 一、为什么这些操作会成为性能瓶颈

我们逐个拆开看：

---

## ① Handle 对象创建成本

每次 Load 都会创建：

```
AsyncOperationHandle struct
+ 内部 Operation 对象（class）
```

实际上创建的不只是一个结构体，而是一整套对象链：

```
Handle
 └── AsyncOperationBase
        ├── TaskCompletionSource
        ├── CallbackList
        ├── DependencyHandles
```

这意味着一次 Load 至少会产生：

- 堆内存分配
    
- GC压力
    
- 内部初始化逻辑
    

如果你一帧发起 100 次 Load：

👉 就是 100 次对象构造 + GC 压力

而加载一个已经缓存的小 Sprite 可能只需要：

```
一次字典查找
```

对比：

|操作|耗时级别|
|---|---|
|字典查找|纳秒级|
|对象构造|微秒级|
|GC触发|毫秒级|

---

## ② Task 调度成本

Addressables 的 Async API 本质基于：

```
Task + 状态机
```

当你调用：

```csharp
await Addressables.LoadAssetAsync<T>();
```

实际发生：

```
创建 Task
注册 continuation
加入调度队列
主线程轮询
恢复执行
```

即使资源已经加载完成，也仍然会走这一整套流程。

换句话说：

> 就算资源早就在内存里，async 机制仍然要跑完整套调度流程

这就是为什么高频加载小资源会慢。

---

## ③ 回调执行成本

每个 Handle 都会维护回调列表：

```
Completed += callback
```

加载完成时：

```
遍历 callback list
逐个调用
```

如果你同时触发 200 个加载：

```
完成时 = 执行200次回调
```

回调本身可能很轻，但调用本身就是成本。

---

# 二、为什么小资源场景最明显

这是很多人没意识到的重点：

> 资源越小，调度成本占比越大

举例：

---

### 加载一个 5MB Texture

耗时分布：

```
IO读取        80%
解压          15%
调度           5%
```

---

### 加载一个已缓存 Sprite

耗时分布：

```
调度           70%
回调           20%
查缓存         10%
```

所以才会出现一种现象：

> 小资源加载次数越多，性能越差

不是资源慢，而是调度慢。

---

# 三、Unity 官方为什么仍然使用这种设计

因为 Addressables 的目标不是极限性能，而是：

```
通用性 + 可扩展性 + 安全性
```

它必须支持：

- 远程下载
    
- 依赖链加载
    
- Bundle加载
    
- 异步IO
    
- 进度监控
    
- 失败重试
    
- 取消操作
    

为了支持这些能力，它必须使用：

> 通用异步任务系统

而通用系统的代价就是：

> 额外开销

---

# 四、为什么成熟项目一定要做加载请求合并

因为合并后：

```
100次 Load → 1次真实 Load + 100次引用
```

性能变化：

|阶段|未合并|合并后|
|---|---|---|
|Handle创建|100|1|
|Task创建|100|1|
|回调调度|100|1|

性能提升：

> 可能是数量级级别（×50~×200）

---

# 五、真实项目中的典型性能灾难场景

以下代码极其常见：

```csharp
foreach(var item in itemList)
{
    icon = await Addressables.LoadAssetAsync<Sprite>(item.icon);
}
```

如果列表 200 个：

```
创建200个Task
创建200个Handle
注册200个回调
```

这比加载图本身还慢。

---

# 六、行业级解决方案（标准做法）

成熟项目不会允许 UI 直接 Load：

统一改成：

```
UI → AssetSystem.Load
```

内部逻辑：

```
如果正在加载 → await同一个Task
如果已加载 → 直接返回
否则 → 真正Load
```

这叫：

> **请求去重（Request Coalescing）**

这是资源系统性能分水岭功能。

---

# 七、一句话本质总结（建议记住）

> Addressables 的加载成本分两类  
> 大资源 → IO成本主导  
> 小资源 → 调度成本主导

---

# 🏆 最终专业级理解总结

Addressables 慢的不是加载，而是：

> **调度系统的通用性成本**

这不是缺陷，而是设计取舍。

---

✅ **给你一个判断经验（非常实用）**

当你发现：

```
资源加载很快
但整体加载流程很慢
```

90%概率问题在：

```
加载调用次数过多
```

而不是资源本身。
