Unity 的 **GUID（Globally Unique Identifier，全局唯一标识符）** 是 Unity 用来唯一标识资源文件的一串字符串，用于在工程内部追踪资源引用关系，而不是靠文件名或路径。

---

## 一、GUID 本质是什么

每个 Unity 资源文件（如 prefab、材质、脚本、贴图等）旁边都会有一个 `.meta` 文件，例如：

```
Player.prefab
Player.prefab.meta
```

打开 `.meta` 文件会看到类似内容：

```
guid: 3f8c1c9a2f6b44c9b0d65f7d8e8e6abc
```

这串 **32位十六进制字符串** 就是 GUID。

👉 Unity 内部引用资源时，用的不是路径，而是这个 GUID。

---

## 二、为什么需要 GUID（核心作用）

如果没有 GUID，Unity 的引用系统会非常脆弱。

假设场景引用了一个材质：

如果只靠路径引用：

```
Assets/Materials/M.mat
```

当你：

- 改文件名
    
- 移动文件夹
    
- 重构目录
    

路径变了 → 引用全部丢失 ❌

---

而 GUID 机制：

引用方式类似：

```
引用 = GUID: 3f8c1c9a...
```

只要 `.meta` 文件没变：

- 文件改名 ✔
    
- 移动目录 ✔
    
- 重构工程 ✔
    

引用都不会丢失 ✅

---

## 三、GUID 的关键规则（非常重要）

### 1️⃣ GUID 存在于 `.meta` 文件

不是存在资源文件本身。

👉 所以版本控制一定要提交 `.meta` 文件

否则：

- 别人拉代码时会生成新 GUID
    
- 所有引用断裂 💥
    

---

### 2️⃣ GUID 一旦生成就不会变

除非：

- 删除 meta 文件
    
- 手动修改 meta
    
- 重新导入资源
    

---

### 3️⃣ GUID 冲突会导致灾难

如果两个资源 GUID 一样：

Unity 会认为它们是同一个资源

结果：

- 引用错乱
    
- Prefab 丢失引用
    
- Script 指向错误
    

常见原因：

- 手动复制文件 + meta
    
- Git 冲突误改
    

---

## 四、Unity 内部引用是怎么存的

Unity YAML 序列化文件中引用长这样：

```yaml
m_Material:
  fileID: 2100000
  guid: 3f8c1c9a2f6b44c9b0d65f7d8e8e6abc
  type: 2
```

Unity 通过：

```
guid + fileID
```

唯一定位资源中的对象。

---

## 五、开发中必须知道的实际经验

这是工作中最容易踩坑的点：

### ✔ 正确迁移资源方法

要带着 `.meta` 文件一起移动。

正确方式：

```
用 Unity Editor 移动
```

不要：

```
用资源管理器移动 ❌
```

---

### ✔ Git 必读规则

`.gitignore` 必须包含：

```
!*.meta
```

否则整个项目引用会炸。

---

### ✔ 判断引用丢失的原因

如果 Prefab 丢失引用：  
优先检查：

> 对应资源 meta 是否变了

---

## 六、一句话理解（面试级总结）

> Unity 的 GUID 是资源的唯一身份证号，Unity 内部所有引用关系都通过 GUID 建立，而不是文件路径。

---

## 七、进阶理解（架构层面）

Unity 的资源系统其实是：

```
文件系统 = 人类看的
GUID系统 = 引擎看的
```

路径只是 UI 表现层  
GUID 才是真实数据层
