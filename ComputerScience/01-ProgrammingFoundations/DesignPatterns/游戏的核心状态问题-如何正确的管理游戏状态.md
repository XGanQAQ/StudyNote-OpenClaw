
## 使用有限状态机来管理全局游戏状态

游戏开发经常会出现比如说，开始游戏前，开始游戏，游戏中，游戏结束这些状态，那我该如何用代码编写这些状态之间的切换和管理？

---

在游戏里常见的 **“开始游戏前 → 开始游戏 → 游戏中 → 游戏结束”** 这类流程，本质就是 **有限状态机（FSM）**。下面给出一套**可直接拷进 Unity 项目**的通用写法：  
1. 纯 C# 版（无 MonoBehaviour 依赖，方便单元测试）  
2. 状态切换示例  
3. 生命周期钩子（Enter / Tick / Exit）  
4. 与事件总线、UI、网络层的衔接方式  

------------------------------------------------  
1. 定义状态枚举 & 事件

```csharp
public enum GameState
{
    PreGame,    // 大厅/等待
    Starting,   // 开始倒计时
    Running,    // 真正战斗
    Ending      // 结算/回放
}

// 触发状态机推进的事件
public readonly struct GameStartRequested;
public readonly struct GameEndRequested;
```

------------------------------------------------  
2. 状态基类（无 MonoBehaviour）

```csharp
public abstract class State
{
    public abstract void Enter();
    public abstract void Tick(float dt);
    public abstract void Exit();
}
```

------------------------------------------------  
3. 具体状态实现示例

```csharp
public sealed class PreGameState : State
{
    public override void Enter()
    {
        // 打开大厅 UI
        UIManager.I.ShowLobby();
        // 注册“开始按钮被按下”
        EventBus.Subscribe<GameStartRequested>(OnStart);
    }

    public override void Tick(float dt) { /* 可做大厅逻辑 */ }

    public override void Exit()
    {
        UIManager.I.CloseLobby();
        EventBus.Unsubscribe<GameStartRequested>(OnStart);
    }

    private void OnStart(GameStartRequested _) =>
        GameStateMachine.I.ChangeState(GameState.Starting);
}
```

------------------------------------------------  
4. 状态机管理器（单例，无 MonoBehaviour）

```csharp
public sealed class GameStateMachine
{
    public static GameStateMachine I { get; } = new();
    private GameStateMachine() { }

    private readonly Dictionary<GameState, State> _states = new()
    {
        { GameState.PreGame,   new PreGameState() },
        { GameState.Starting,  new StartingState() },
        { GameState.Running,   new RunningState() },
        { GameState.Ending,    new EndingState() }
    };

    private State _current;
    public GameState CurrentType { get; private set; }

    public void ChangeState(GameState newType)
    {
        _current?.Exit();
        CurrentType = newType;
        _current = _states[newType];
        _current.Enter();
    }

    // 由外部每帧调用，可放在 NetworkLoop 或 MonoBehaviour 的 Update
    public void Tick(float dt) => _current?.Tick(dt);
}
```

------------------------------------------------  
5. 与 Unity 生命周期衔接

在任意 MonoBehaviour 里驱动：

```csharp
public class GameLoop : MonoBehaviour
{
    void Start()  => GameStateMachine.I.ChangeState(GameState.PreGame);
    void Update() => GameStateMachine.I.Tick(Time.deltaTime);
}
```

------------------------------------------------  
6. 触发状态切换的 3 种常见方式

| 触发源 | 示例代码 |
|---|---|
| UI 按钮 | `EventBus.Raise(new GameStartRequested());` |
| 网络消息 | `NetworkClient.OnGameStart += _ => GameStateMachine.I.ChangeState(GameState.Running);` |
| 脚本逻辑 | 在 `RunningState.Tick()` 内检测失败条件，然后 `GameStateMachine.I.ChangeState(GameState.Ending);` |

------------------------------------------------  
7. 如何扩展

• 需要“子状态” → 在 `RunningState` 里再嵌一个子 FSM（例如战斗阶段的 `Idle → Reload → Fire`）。  
• 需要异步倒计时 → 在 `StartingState.Enter()` 里启动 `async UniTaskVoid CountdownAsync()`。  
• 需要断线重连 → 把 `GameStateMachine.I.ChangeState()` 改为可带参数的结构体，支持恢复上下文。

------------------------------------------------  
一句话总结  
把“阶段”封装成 `State`，用 `GameStateMachine` 统一调度 **Enter / Tick / Exit** 三大钩子；  
任何系统（UI、网络、玩法）只需调用 `ChangeState` 或抛事件即可推进流程，**无需关心其他模块细节**。

## 使用状态机发布事件 让各个系统模块得到游戏全局状态信息

游戏流程的推进常常会根据游戏的状态，我该如何让各个游戏模块得到游戏的状态信息？我过去都是用单例去之间传递状态信息，但是我想知道最正确最合适的方法

---

不要再让各模块去“读”单例里的状态，而是让 **状态机把“状态变化”主动推送给所有关心的人**。  
核心做法就是：**“状态机发布事件 → 各模块订阅事件”**。这样既解耦，又天然支持“谁关心谁订阅”，不会漏通知、也不会读到脏数据。

下面给出一条 **最小可落地的链路**（可直接替换你现在的单例读状态）：

------------------------------------------------
1. 状态机只负责两件事  
   • 记录当前状态枚举  
   • 每次切换时 **广播事件**

```csharp
public sealed class GameStateMachine
{
    public static GameStateMachine I { get; } = new();
    private GameState _current;

    public void ChangeState(GameState newState)
    {
        var old = _current;
        _current = newState;

        // 1️⃣ 推送事件
        EventBus.Raise(new GameStateChanged(old, newState));
    }

    public GameState Current => _current;   // 仅供调试
}
```

2. 事件定义（纯数据，零引用）

```csharp
public readonly struct GameStateChanged
{
    public readonly GameState From;
    public readonly GameState To;
    public GameStateChanged(GameState f, GameState t) { From = f; To = t; }
}
```

------------------------------------------------
3. 各模块如何“得到状态信息”

• **UI 系统**  
```csharp
public class MainUI : MonoBehaviour
{
    void OnEnable () => EventBus.Subscribe<GameStateChanged>(OnState);
    void OnDisable() => EventBus.Unsubscribe<GameStateChanged>(OnState);

    void OnState(GameStateChanged e)
    {
        switch (e.To)
        {
            case GameState.PreGame:  ShowLobby(); break;
            case GameState.Running:  HideLobby(); break;
            case GameState.Ending:   ShowResult(); break;
        }
    }
}
```

• **音频系统**  
```csharp
EventBus.Subscribe<GameStateChanged>(e =>
{
    if (e.To == GameState.Running) AudioManager.I.PlayMusic("battle");
});
```

• **网络同步层**  
```csharp
EventBus.Subscribe<GameStateChanged>(e =>
{
    if (e.To == GameState.Ending)
        NetworkClient.SendGameOver();
});
```

------------------------------------------------
4. 如果某模块确实只想知道“当前状态”而不关心变化

给它一个 **只读接口**，仍然不暴露可写字段：

```csharp
public interface IReadOnlyGameState
{
    GameState Current { get; }
}
```

状态机实现该接口，模块通过依赖注入 / 服务定位器拿只读引用：

```csharp
public class SomeSystem
{
    public SomeSystem(IReadOnlyGameState state) { _state = state; }
    void Tick()
    {
        if (_state.Current == GameState.Running) DoSomething();
    }
}
```

------------------------------------------------
5. 这样做的好处

| 老做法（单例读状态） | 新做法（事件 + 只读接口） |
|---|---|
| 模块必须知道单例存在 → 耦合 | 只关心事件或接口 → 解耦 |
| 易读到脏/过期状态 | 永远收到“原子”的状态变化事件 |
| 不易扩展（新增状态要改所有模块） | 新增模块只需订阅事件 |
| 单元测试难（静态字段） | 可注入 Mock 状态机 |

------------------------------------------------
一句话总结  
**“状态机是事件的唯一发布者，各模块只订阅事件或拿到只读接口。”**  
这样就把“状态信息”从 **被拉取的共享变量** 变成 **被推送到门口的不可变消息**，彻底摆脱单例全局状态的泥潭。

### 利用事件总线

我给出的示例里 **状态机就是事件总线的“发布者”**，它通过

```csharp
EventBus.Raise(new GameStateChanged(old, newState));
```

把“状态切换”这一事件广播出去，其他模块再通过

```csharp
EventBus.Subscribe<GameStateChanged>(OnState);
```

来接收。  
换句话说，状态机 **不直接引用** UI、音频、网络等任何下游系统，而是 **完全依赖事件总线** 完成解耦。
