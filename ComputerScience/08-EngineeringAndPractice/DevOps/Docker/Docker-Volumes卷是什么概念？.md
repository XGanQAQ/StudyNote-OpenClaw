
## 🧩 一、Docker 容器的数据本质

默认情况下，Docker 容器内部的数据是：

> 写在容器的**临时层**里（叫「容器层」）。

这层数据在：

- 删除容器；
    
- 重新构建镜像；
    
- 或重启 `docker-compose up` 时；
    

都会被删除。

换句话说：

> ❌ **容器本身的数据是临时的，不持久化。**

---

## 📦 二、Volume 是什么？

可以把 **volume（卷）** 理解成：

> 💾 「容器外的一块持久化磁盘」，  
> 用来保存你想在容器销毁后仍然保留的数据。

这块“磁盘”可以是：

- Docker 自己管理的一块存储空间（匿名卷 / 命名卷），或
    
- 你手动映射的宿主机目录（bind mount）。
    

---

## 🧱 三、举个直观的例子

假设你运行一个数据库容器：

```bash
docker run -d --name db postgres
```

如果你往数据库里加了数据，然后删掉容器：

```bash
docker rm -f db
```

⚠️ 数据全没了。因为它都在容器的临时层里。

---

现在我们给它挂载一个卷：

```bash
docker run -d --name db \
  -v my-postgres-data:/var/lib/postgresql/data \
  postgres
```

这时：

- `my-postgres-data` 是 Docker 创建的卷；
    
- 数据会保存在这块卷里；
    
- 即使容器被删除，卷的数据还在。
    

你可以重新运行容器挂载同一个卷：

```bash
docker run -d --name new-db \
  -v my-postgres-data:/var/lib/postgresql/data \
  postgres
```

💡 数据依然存在，因为卷在容器外部。

---

## ⚙️ 四、在 docker-compose 中的使用

例如你现在的配置：

```yaml
volumes:
  - ./gitea:/data
  - ./postgres:/var/lib/postgresql/data
```

这两行的意思是：

> “请把宿主机当前目录下的 `./gitea` 文件夹，映射到容器内的 `/data` 路径。”

于是：

- 宿主机的 `./gitea` 就像「外接硬盘」；
    
- 容器在 `/data` 里写的内容，其实都存储在宿主机上；
    
- 删除容器不会影响宿主机的 `./gitea` 文件。

## 🚀 五、Volume vs 不使用 Volume 的区别总结

|项目|不使用卷|使用卷（或挂载目录）|
|---|---|---|
|数据保存位置|容器内部（临时层）|容器外部（持久化存储）|
|删除容器后|数据消失|数据仍在|
|容器迁移|数据随容器消失|数据独立存在，可被新容器复用|
|性能|快（但易丢失）|稍慢（但安全可靠）|
|常见用途|测试镜像、一次性容器|数据库、网站、仓库等需持久化场景|

---

## 🧰 六、三种卷类型总结

|类型|写法|特点|
|---|---|---|
|匿名卷|`-v /data`|Docker 自动生成名字，删除容器不会删卷|
|命名卷|`-v mydata:/data`|可被多个容器共享|
|绑定挂载（Bind Mount）|`-v ./data:/data`|显式映射宿主机目录，最常用（你就是这个）|

---

## 🧠 七、通俗比喻总结

- 🏠 **容器**：租的公寓（临时的）
    
- 💾 **Volume（卷）**：外接硬盘（不随公寓销毁）
    
- 🪣 **Bind Mount（挂载宿主机目录）**：你直接把自己的电脑硬盘拿来用
    
- 🧨 **不使用卷**：容器一删，资料全没；重建相当于全新系统
    
- 🔒 **使用卷**：容器一删，硬盘还在，换个容器还能继续用
    

---

## ✅ 八、总结一句话

> Docker 的 volume 是一种 **容器外的数据持久化机制**，  
> 让你在删除、重启或迁移容器时，数据不会丢失。  
> 它的核心作用是「数据与容器生命周期解耦」。
